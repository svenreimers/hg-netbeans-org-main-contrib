Bugs and Limitations in the Debugger Integration

  • Once launched, there is no integration, so the GUI is that of the original debugger implementation. In particular, the source browser does not use the NetBeans editor.
  • When debugging a JUnit test, the debugger will prompt you for JUnit sources to display. Just click Cancel, or download JUnit sources and associate them with the junit library in Library Manager.
  • You must build the project before debugging. If you have not, the IDE will display a warning.
  • The .debuggerDefaults is currently overwritten each time you run the debugger, so if you have some important customizations it is better to run omnidebug.xml directly after the first time.
  • The debugger supports an offline instrumentation mode but the NetBeans integration does not currently take advantage of it to speed up debugger launching.
  • Closing the debugger window does not stop the process; you must use File | Exit. If you forget, choose Build | Stop Build/Run from the IDE.
  • Passing system properties when debugging a test case may not work reliably for NetBeans module projects. Try defining basic properties like nb_all or netbeans.dest.dir in nbproject/private/private.properties.
  • If your program expects a particular working directory to be set when it runs, it will not work correctly in the debugger, since that is always launched with the working directory set to nbproject/private/cache/.
